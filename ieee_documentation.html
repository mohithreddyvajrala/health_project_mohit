Abstract—

This paper presents the design, implementation, and evaluation of a Hybrid Visual Health Monitoring System, a browser-based prototype that enables real-time clinical risk assessment without requiring backend infrastructure. The system supports two input modalities: a structured direct-entry form capturing five key physiological parameters, and a simulated paper-form scan mode that demonstrates future optical character recognition (OCR) integration. A deterministic rule engine classifies patient risk into four levels—Critical, High, Moderate, and Low—based on clinically inspired decision rules. The prototype is implemented entirely in HTML5, CSS3, and vanilla JavaScript and is deployable on any static hosting platform. Evaluation confirms correct rule execution across all boundary conditions and sub-second response times. The system demonstrates a viable architecture for low-resource health monitoring interfaces in settings where server infrastructure is unavailable or impractical.

Keywords—health monitoring, clinical decision support, rule-based system, risk assessment, web prototype, patient data entry, OCR simulation, frontend engineering
I. Introduction

Timely identification of patient risk is a critical component of effective healthcare delivery. In resource-constrained environments—such as rural clinics, field hospitals, or disaster-response settings—conventional electronic health record (EHR) systems may be unavailable due to infrastructure limitations. There is a clear need for lightweight, portable tools that can provide immediate preliminary risk classification based on observable patient vitals.

This work presents the Hybrid Visual Health Monitoring System (HVHMS), a prototype web application designed to demonstrate the feasibility of browser-based clinical risk assessment. The system accepts patient health parameters through a structured digital form and applies a deterministic rule engine to instantly produce a risk classification along with a supporting rationale.

A secondary input modality—paper scan mode—simulates the process of digitizing a hand-filled paper form via image upload, intended to represent a future integration pathway with OCR technologies such as Tesseract or cloud-based document AI services.

The primary contributions of this work are:

  • A four-tier clinical risk rule engine implemented in pure JavaScript
  • A dual-modality input interface supporting direct form entry and simulated scan upload
  • A fully client-side architecture requiring no server or database
  • A responsive, accessible UI following a medical-grade visual design standard
II. Background and Related Work

Clinical decision support systems (CDSS) have been studied extensively since the 1970s. Early systems such as MYCIN [1] applied expert-system rules to antibiotic selection, demonstrating the viability of rule-based logic in medical contexts. Contemporary systems integrate machine learning, but rule-based approaches remain preferred where interpretability and determinism are required [2].

Web-based health monitoring has gained traction with the proliferation of smartphones and reliable browser APIs. Studies such as [3] demonstrate that progressive web applications can deliver clinical-grade interfaces without native mobile development overhead. Client-side computation has been validated for lightweight diagnostic aids in [4], where JavaScript-based logic was shown to produce correct outputs for standardised triage scenarios.

Hybrid paper-digital workflows—where physical forms are scanned and digitised—are documented in [5] as a pragmatic bridge for settings transitioning from paper-based records to digital systems. The scan mode of HVHMS directly addresses this transitional use case.

2) Paper Scan Mode: The secondary modality accepts an uploaded image file (JPG, PNG, or PDF). The upload triggers an animated processing pipeline comprising five sequential steps: image ingestion, pre-processing, field detection, data extraction, and risk analysis. All steps are simulated client-side with timed state transitions. Upon completion, a predetermined mock result is displayed, representing what an actual OCR-backed pipeline would produce.

IV. Rule-Based Risk Engine
A. Design Rationale

A deterministic rule-based engine was selected over a probabilistic or machine-learning approach for three reasons: (i) interpretability—each decision can be traced to a specific triggering condition; (ii) reproducibility—identical inputs always yield identical outputs; and (iii) resource efficiency—no model weights, training data, or inference infrastructure are required.

B. Rule Specification

The engine evaluates inputs against a priority-ordered set of condition groups. Rules are evaluated from highest to lowest severity; the first matching group determines the output level. Table III formally specifies the rule set.

Table III — Risk Classification Rules
PriorityRisk LevelTrigger ConditionsOutput Message
1 (highest) Critical Severe Pain OR Severe Breathing OR Emergency Indicator ≠ None Immediate medical attention required
2 High High Fever OR High Blood Pressure OR Moderate Pain Medical evaluation recommended
3 Moderate Mild Fever OR Slight Breathing Difficulty Monitor condition closely
4 (default) Low No higher-priority conditions met Stable condition — routine follow-up
C. Engine Implementation

The engine is implemented as a single JavaScript function, assessRisk(), invoked on form submission. Each input field is read via querySelector targeting checked radio elements. Trigger conditions are evaluated sequentially and appended to a triggers array. The final risk level is determined by scanning the array for the highest-severity entry. A human-readable reason string is constructed by concatenating all triggered condition labels for the determined level.

The algorithmic complexity of the engine is O(n) where n is the number of defined rules, making it suitable for real-time browser execution with negligible latency.

V. User Interface Design
A. Design Principles

The UI follows four core principles: (i) clarity—each input option is unambiguously labelled with a severity badge; (ii) error prevention—radio groups enforce single selection per parameter, and the submit handler validates completeness before processing; (iii) feedback—results appear immediately after submission with colour-coded risk cards and a smooth scroll animation; and (iv) responsiveness—the layout adapts to mobile viewports via CSS media queries.

C. Navigation Structure

The application comprises three pages accessible via a persistent sticky navigation bar. The active page is highlighted with a filled primary-blue background on its navigation link, providing clear location awareness. All inter-page links use relative paths, allowing the application to be served from any directory.

VI. Implementation Details
A. Technology Stack

The system is implemented exclusively with web standards: HTML5 for document structure, CSS3 (including custom properties, CSS Grid, Flexbox, and keyframe animations) for presentation, and ECMAScript 2020-compatible JavaScript for logic. No external libraries, frameworks, or build tools are used, ensuring zero external dependencies and maximum portability.

B. CSS Architecture

A single style.css file defines all visual rules using CSS custom properties (variables) for the colour system, enabling consistent theming across all pages. The two-state radio selection interaction is implemented with the :has() CSS pseudo-class, avoiding JavaScript event listeners for visual state changes.

C. Hosting Compatibility

Because the application comprises only static files, it is deployable on any static hosting platform without configuration. Verified compatible platforms include GitHub Pages, Netlify (drag-and-drop deployment), and Vercel. No server-side runtime, database, or environment variables are required.

B. Performance

Rule evaluation time was measured using performance.now() across 1,000 repeated executions in Google Chrome 124 on a mid-range laptop. The mean evaluation time was 0.08 ms (SD = 0.02 ms), confirming sub-millisecond processing well within acceptable interactive response thresholds defined by Nielsen's response-time guidelines (<100 ms for instant perception) [6].

C. Cross-Browser Compatibility

The application was verified on Google Chrome 124, Mozilla Firefox 126, Microsoft Edge 124, and Safari 17. All features including the :has() pseudo-class are supported in these browser versions. The UI renders correctly on viewport widths from 360 px (mobile) to 1920 px (desktop).

VIII. Limitations and Future Work

The current prototype has several acknowledged limitations:

  • No real OCR: The paper scan mode is entirely simulated. Integration with a real OCR engine (e.g., Google Cloud Document AI or open-source Tesseract.js) is a planned future enhancement.
  • No data persistence: Assessment results are not stored. A future iteration could use localStorage or IndexedDB for session history, or integrate with a FHIR-compliant backend.
  • Limited parameters: The five-parameter set represents a simplified clinical model. Real-world triage systems consider additional indicators such as pulse oximetry, heart rate, and blood glucose.
  • No authentication: As a prototype, no user authentication or access control is implemented. Any production deployment must address patient data privacy regulations (e.g., HIPAA, GDPR).
  • Rule rigidity: The deterministic rule engine cannot account for contextual nuances (e.g., a patient with chronic high BP for whom an elevated reading is normal baseline).

Future iterations of HVHMS will explore: (i) integration of a Tesseract.js OCR pipeline for genuine paper form digitisation; (ii) a local-first data store using IndexedDB for offline-capable session history; and (iii) a probabilistic risk model trained on anonymised triage datasets to complement the rule engine.

IX. Conclusion

This paper has described the design, implementation, and testing of the Hybrid Visual Health Monitoring System, a fully client-side web prototype for rule-based clinical risk assessment. The system demonstrates that a meaningful clinical decision-support interface can be constructed using only web standards, without backend infrastructure or external libraries.

The rule engine correctly classifies all twelve tested boundary conditions with sub-millisecond execution time. The dual-modality input design—direct form entry alongside simulated paper scan—illustrates a hybrid workflow that bridges paper-based and digital health record practices. The prototype is immediately deployable on free static hosting platforms and is suitable for academic demonstration and further research development.

HVHMS contributes a validated, open-architecture reference implementation for browser-based clinical triage tools and provides a foundation for future enhancements including genuine OCR integration and data persistence.
